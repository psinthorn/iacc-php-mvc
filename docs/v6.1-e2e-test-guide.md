## 2. Happy path (5 min) — enqueue, run, see it land

Goal: verify worker pickup, dispatch, success recording, and dashboard auto-refresh.

```bash
# Enqueue a known-good echo task for company_id=1
docker exec iacc_mysql mysql -uroot -proot iacc -e "
INSERT INTO task_queue (company_id, task_type, payload, priority, status)
VALUES (1, 'echo', '{\"msg\":\"hello v6.1\"}', 5, 'pending');"

# Refresh dashboard → row appears in "Pending/Running" column with priority badge
# Wait up to 30s for auto-refresh OR force refresh

# Run the worker once (CLI)
docker exec iacc_php php /var/www/html/scripts/worker.php

# Refresh dashboard → row should now be GONE from Pending (status='done', filtered out)
# Or: query DB directly
docker exec iacc_mysql mysql -uroot -proot iacc \
  -e "SELECT id, status, attempts, last_error FROM task_queue ORDER BY id DESC LIMIT 1;"
```

**Pass criteria**:
- Pending row visible in dashboard with correct priority badge color (5 = blue/normal)
- Click **Details** → modal opens via fetch(), shows payload pretty-printed, status badge, attempt counter
- After worker run: status='done', attempts=1, last_error=NULL
- One row in `task_results` with `success=1`, duration_ms > 0
- Auto-refresh fires every 30s (watch the network tab — request to current page repeats)

## 3. Retry + dead-letter path (5 min)

Goal: verify exponential backoff (60s/300s/1500s/7200s), max-attempts enforcement, dead-letter promotion.

```bash
# Enqueue a deliberately-failing echo task
docker exec iacc_mysql mysql -uroot -proot iacc -e "
INSERT INTO task_queue (company_id, task_type, payload, priority, status, max_attempts)
VALUES (1, 'echo', '{\"fail\":true}', 3, 'pending', 3);"
TASK_ID=$(docker exec iacc_mysql mysql -uroot -proot iacc -Nse \
  "SELECT id FROM task_queue ORDER BY id DESC LIMIT 1;")
echo "Task ID: $TASK_ID"

# Run worker — should fail attempt 1 of 3
docker exec iacc_php php /var/www/html/scripts/worker.php
docker exec iacc_mysql mysql -uroot -proot iacc \
  -e "SELECT status, attempts, scheduled_for, last_error FROM task_queue WHERE id=$TASK_ID;"
# Expect: status=pending, attempts=1, scheduled_for ≈ NOW()+60s, last_error="forced failure"

# Speed up backoff to test attempt 2 without waiting 60s:
docker exec iacc_mysql mysql -uroot -proot iacc \
  -e "UPDATE task_queue SET scheduled_for=NOW() WHERE id=$TASK_ID;"
docker exec iacc_php php /var/www/html/scripts/worker.php
# attempts=2, scheduled_for ≈ NOW()+300s

# Repeat for attempt 3 → should now go to dead_letter
docker exec iacc_mysql mysql -uroot -proot iacc \
  -e "UPDATE task_queue SET scheduled_for=NOW() WHERE id=$TASK_ID;"
docker exec iacc_php php /var/www/html/scripts/worker.php
docker exec iacc_mysql mysql -uroot -proot iacc \
  -e "SELECT status, attempts FROM task_queue WHERE id=$TASK_ID;"
# Expect: status=dead_letter, attempts=3
```

**Pass criteria**:
- Each failed attempt creates a row in `task_results` with `success=0` and `error_message` populated
- Backoff schedule increments: 60s → 300s → 1500s (per RetryPolicy::DEFAULT_BACKOFF)
- After max_attempts hit, row promotes to `dead_letter` and stops being claimed
- Dashboard's "Failed/Dead-Letter" column shows the row in red, with truncated error preview

## 4. Bulk actions (3 min)

Goal: verify Bulk Requeue + Bulk Clear with the 500-row cap and CSRF.

```bash
# Seed 5 dead-letter rows for company 1
docker exec iacc_mysql mysql -uroot -proot iacc -e "
INSERT INTO task_queue (company_id, task_type, payload, status) VALUES
  (1, 'echo', '{}', 'dead_letter'),
  (1, 'echo', '{}', 'dead_letter'),
  (1, 'echo', '{}', 'dead_letter'),
  (1, 'echo', '{}', 'dead_letter'),
  (1, 'echo', '{}', 'dead_letter');"
```

In the dashboard's "Failed/Dead-Letter" column:
1. Click **Bulk Requeue** → confirm dialog mentions "up to 500 rows"
2. Confirm → flash message shows N requeued
3. Refresh → all 5 should now be in Pending column (status='pending', attempts=0)
4. Reseed 5 dead-letters, click **Bulk Clear** → flash shows N soft-deleted
5. Refresh → 0 in dead-letter column. DB check: `SELECT COUNT(*) FROM task_queue WHERE status='dead_letter' AND deleted_at IS NULL` → 0

**Pass criteria**:
- Bulk forms carry a CSRF token (inspect HTML)
- Submitting without token → 403
- 500-row cap enforced server-side (model layer, not just UI) — confirmed by `tmp/dashboard-test.php` test 12

## 5. Multi-tenant isolation (5 min) — CRITICAL

Goal: prove cross-tenant access is blocked at the model layer.

```bash
# Seed a task for "other" tenant
docker exec iacc_mysql mysql -uroot -proot iacc -e "
INSERT INTO task_queue (company_id, task_type, payload, status)
VALUES (999, 'echo', '{\"OTHER_TENANT\":true}', 'pending');"
OTHER_ID=$(docker exec iacc_mysql mysql -uroot -proot iacc -Nse \
  "SELECT id FROM task_queue WHERE company_id=999 ORDER BY id DESC LIMIT 1;")

# Run the model test suite (covers 21/21 tenant guards)
docker exec iacc_php php /var/www/html/tmp/dashboard-test.php
# Expect: 21 passed, 0 failed
```

In the browser (logged in as company_id=NULL super-admin):
1. Dashboard should show only your own company's rows (or all if super-admin scope is enabled)
2. Try URL-tampering: `index.php?page=admin_task_queue_details&id=$OTHER_ID`
   - **Pass**: 404 / "task not found" — controller never returns the other tenant's row
3. Try POST to `admin_task_queue_retry` with `task_id=$OTHER_ID` — should fail with no-op

**Pass criteria** (this is the deal-breaker — failing here blocks release):
- No URL/parameter tampering can leak another tenant's data
- Every model method takes `company_id` as first arg and filters every query
- `findWithHistory(WRONG_company, real_id)` returns null
- `softDelete(WRONG_company, real_id)` returns false (no row affected)

## 6. Worker crash recovery (3 min)

Goal: verify the 10-minute stale-lock reaper.

```bash
# Manually create a "stuck" locked row (simulates a worker that crashed mid-task)
docker exec iacc_mysql mysql -uroot -proot iacc -e "
INSERT INTO task_queue (company_id, task_type, payload, status, locked_at, locked_by)
VALUES (1, 'echo', '{}', 'locked',
        NOW() - INTERVAL 15 MINUTE,
        'cpanel-php-DEAD-1234');"

# Run worker — should reap the stale lock back to 'pending' and pick it up
docker exec iacc_php php /var/www/html/scripts/worker.php

# Verify
docker exec iacc_mysql mysql -uroot -proot iacc \
  -e "SELECT status, attempts, locked_at, locked_by FROM task_queue ORDER BY id DESC LIMIT 1;"
# Expect: status='done' (reaped → claimed → ran), locked_at/locked_by cleared
```

**Pass criteria**:
- A locked row >10 min old is automatically returned to pending
- Worker output reports `stale_reaped > 0` for that run
- No orphaned `task_results` row (stale reap doesn't synthesize a fake failure)

## Common gotchas (from sprint #75–#78 development)

- **Worker can't run from cron**: cron token not set locally — use the CLI form above (`scripts/worker.php`) for tests
- **PHP 7.4 in container vs 8.x in IDE**: `match` expression flagged as syntax error in IDE but not really — always verify with `docker exec iacc_php php -l <file>`
- **30s auto-refresh vs modal**: scrolling the page resets the timer; opening modal pauses it (intended)
- **Priority badges**: high (1-3) red, normal (4-6 default 5) blue, low (7-10) green — check [PRIORITY.md](../app/Workers/PRIORITY.md) for the convention
- **Bulk cap**: 500 rows enforced at model layer (`TaskQueue::BULK_ACTION_CAP`); UI confirm() also says 500. Don't lower one without the other.
